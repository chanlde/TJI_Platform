# 光伏清洗设备 OTA 升级方案

## 1. 目标

本文档用于定义光伏清洗设备单片机固件 OTA 升级的完整流程。

OTA 升级的核心目标不是“能把新固件刷进去”，而是：

- 升级前能判断设备是否适合升级
- 下载过程中失败不会影响当前固件运行
- 固件写入后必须校验完整性和合法性
- 新固件启动失败时可以自动回滚旧版本
- 升级成功或失败后，设备可以向 App / 服务器上报明确结果
- 用户可以在 App 上看到升级进度、升级结果和失败原因

### 1.1 产品定位

光伏清洗设备不是消费级高频更新产品，不需要像手机、路由器、智能家居那样频繁推送版本。

本产品 OTA 的定位应该是：

```text
平时不用。
需要修复关键 Bug、适配硬件、优化控制逻辑时再用。
升级必须可控，失败必须能恢复。
```

因此，本方案不追求复杂运营能力，而优先追求：

- 低频升级
- 人工确认
- 稳定可靠
- 失败可回滚
- 现场可判断状态
- 售后可定位原因

第一版 OTA 不建议做自动大规模推送。  
建议采用“App 手动检测更新 + 用户或售后确认升级 + 服务器记录结果”的模式。

## 2. 优先级划分

OTA 功能建议按必要程度分成 4 个等级。

| 优先级 | 定义 | 目标 |
|---|---|---|
| P0 必须做 | 没有这些就不建议上线 OTA | 防止设备变砖 |
| P1 第一版建议做 | 低频商用 OTA 建议具备 | 让用户、现场人员、售后看得懂 |
| P2 后续增强 | 设备量增加或联网公网后再做 | 提升安全性和维护效率 |
| P3 暂不优先 | 对低频更新产品不是刚需 | 降低流量成本或做复杂运维 |

### 2.1 P0 必须做

这些是 OTA 的底线能力。缺少其中任何一项，都可能导致设备远程升级后无法恢复。

| 功能 | 必要性 | 原因 |
|---|---|---|
| Bootloader | 必须 | 负责启动选择和失败回滚 |
| A/B 双固件分区 | 必须 | 新固件坏了还能切回旧固件 |
| OTA 状态区 | 必须 | 记录当前分区、目标分区、升级状态、失败原因 |
| 固件大小校验 | 必须 | 防止下载不完整的固件被启动 |
| 固件 Hash 校验 | 必须 | 防止固件内容损坏 |
| 新固件启动确认 | 必须 | 新固件必须主动报平安，才算升级成功 |
| 看门狗 | 必须 | 新固件死机时能自动复位 |
| 启动失败自动回滚 | 必须 | 防止新固件反复死机导致设备不可用 |
| 升级前设备状态检查 | 必须 | 防止设备工作中升级造成危险 |
| 基础进度上报 | 必须 | App 和服务器要知道升级卡在哪一步 |

P0 的最低目标：

```text
新固件下载失败，不影响旧固件。
新固件校验失败，不启动新固件。
新固件启动失败，自动回滚旧固件。
旧固件恢复后，能上报失败原因。
```

### 2.2 P1 第一版建议做

这些能力不是防变砖的绝对底线，但第一版商用建议一起做，否则后期排查问题会很痛苦。

| 功能 | 建议程度 | 原因 |
|---|---|---|
| LED 升级状态提示 | 强烈建议 | 现场人员能看出设备正在升级还是失败 |
| App 显示升级阶段 | 强烈建议 | 用户知道当前是下载、校验、重启还是验证 |
| 失败原因码 | 强烈建议 | 方便售后和研发定位 |
| 服务器记录升级结果 | 强烈建议 | 后台能统计成功率和失败率 |
| 设备硬件版本匹配 | 强烈建议 | 防止 HW-A 设备刷入 HW-B 固件 |
| 电量/供电检查 | 强烈建议 | 防止升级中断电 |
| 工作状态检查 | 强烈建议 | 清洗中、电机运行中不允许升级 |
| 下载超时重试 | 建议 | 网络差时提高成功率 |
| Flash 写入错误检测 | 建议 | 避免写坏后继续启动 |

P1 的目标：

```text
用户知道升级进行到哪一步。
现场人员能通过 LED 粗略判断状态。
研发能通过失败原因码定位问题。
服务器能看到每台设备升级结果。
```

### 2.3 P2 第二阶段做

这些适合在 OTA 主流程稳定后再做。  
如果设备已经连接公网，HTTPS 和数字签名建议提前。

| 功能 | 价值 | 适合时机 |
|---|---|---|
| HTTPS 下载 | 防止传输被劫持 | 固件通过公网下载时 |
| 数字签名校验 | 防止固件被篡改 | 公网商用前 |
| 版本黑名单 | 禁止继续使用问题版本 | 出现问题版本后 |
| 升级结果统计 | 查看成功率和失败分布 | 设备进入小批量使用后 |
| 小范围试点发布 | 降低批量事故风险 | 一次升级多台设备前 |
| 断点续传 | 网络差时减少重复下载 | 固件较大或网络不稳定时 |
| 失败后再次手动升级 | 方便售后重新尝试 | 失败原因可控后 |

### 2.4 P3 后期优化

这些对低频更新的光伏清洗设备不是刚需，可以长期后置。

| 功能 | 价值 |
|---|---|
| 差分升级 | 减小固件下载体积 |
| 多区域 CDN | 提升大规模下载速度 |
| 多版本自动选择 | 根据设备型号、批次、区域自动匹配固件 |
| 自动批量推送 | 减少人工操作，但风险更高 |
| 强制升级策略 | 对严重 Bug 版本进行强制修复，需谨慎 |
| 防回滚策略 | 防止被恶意刷回存在漏洞的旧版本 |
| 固件加密 | 防止固件被直接读取和逆向 |
| 蓝牙兜底升级 | 网络不可用时现场维护 |
| 工厂产测升级工具 | 方便产线批量刷机 |

### 2.5 推荐实施顺序

建议按下面顺序推进，不要一上来就做复杂后台。

```text
第 1 步：先做 Bootloader + A/B 分区 + OTA 状态区
第 2 步：做固件下载到备用分区
第 3 步：做固件大小和 Hash 校验
第 4 步：做 PENDING_VERIFY 试启动
第 5 步：做新固件启动确认
第 6 步：做启动失败自动回滚
第 7 步：做 App 进度显示和 LED 状态
第 8 步：做服务器版本管理和结果记录
第 9 步：做 HTTPS、签名、小范围试点等增强能力
```

如果资源有限，优先保证第 1 到第 6 步。  
这 6 步决定设备会不会因为坏固件变砖。

对当前产品来说，第一版建议做到：

```text
P0 全部完成。
P1 选择关键项完成。
P2/P3 不作为第一版目标。
```

## 3. 推荐整体架构

建议采用 A/B 双固件分区方案。

```text
Flash 布局示意：

+--------------------------+
| Bootloader               |
+--------------------------+
| App Slot A               |
+--------------------------+
| App Slot B               |
+--------------------------+
| OTA 状态区               |
+--------------------------+
| 设备参数区               |
+--------------------------+
```

各区域职责如下：

| 区域 | 职责 |
|---|---|
| Bootloader | 选择启动哪个固件，校验固件，处理回滚 |
| App Slot A | 一个可运行的 App 固件分区 |
| App Slot B | 另一个可运行的 App 固件分区 |
| OTA 状态区 | 保存升级状态、目标分区、回滚标志、失败原因 |
| 设备参数区 | 保存设备 ID、硬件版本、网络参数、运行配置等 |

当前运行在 A 区时，新固件下载到 B 区。  
当前运行在 B 区时，新固件下载到 A 区。

新固件只有在启动成功、自检通过、并主动写入确认标志后，才算升级成功。否则 Bootloader 应自动切回旧固件。

## 4. 角色分工

### 4.1 服务器

服务器负责：

- 存储不同版本固件
- 管理固件元信息
- 判断设备是否有可升级版本
- 控制哪些设备允许看到新版本
- 接收设备升级进度和升级结果

第一版服务器可以先做简单：

- 上传固件
- 保存版本号、硬件版本、文件大小、Hash、下载地址
- 根据设备版本返回是否有更新
- 接收升级结果
- 支持把某个版本标记为可用或禁用

复杂灰度发布、自动批量推送、统计看板可以后做。  
第一版不建议服务器主动给所有设备推送升级，建议由 App 手动检测，用户或售后确认后再升级。

固件文件建议按硬件版本和固件版本区分：

```text
firmware/
  HW-A/
    v1.0.3/app.bin
    v1.0.4/app.bin
  HW-B/
    v1.0.4/app.bin
```

固件元信息示例（新版服务器接口）：

```json
{
  "id": 8,
  "version": "4",
  "productName": "光伏清洗",
  "techDesc": "光伏清洗bin",
  "innerVersion": "6",
  "publishDate": "2026-05-29 00:00:00",
  "path": "/download/SolarClean_APP_V0.0.1_0X08020000.bin",
  "fileSize": 245760,
  "sha256": "xxxxxxxx",
  "type": 2
}
```

### 4.2 App

App 负责：

- 显示当前固件版本
- 查询是否有新版本
- 显示升级内容
- 让用户确认是否升级
- 下发升级指令
- 显示升级进度
- 显示升级成功、失败、回滚结果

App 不建议直接参与固件分包传输，除非设备没有直接联网能力。  
如果设备具备 Wi-Fi / 4G / Cat.1 / NB-IoT 等联网能力，建议由设备直接从服务器下载固件。

### 4.3 设备 App 固件

设备主固件负责：

- 上报当前版本和设备状态
- 接收 OTA 指令
- 检查升级条件
- 下载新固件
- 写入备用分区
- 校验固件
- 写入 OTA 状态
- 重启进入新固件
- 新固件首次启动后执行自检
- 自检成功后确认升级成功
- 自检失败后主动触发回滚

### 4.4 Bootloader

Bootloader 负责：

- 读取 OTA 状态区
- 判断应该启动 A 区还是 B 区
- 校验待启动固件是否合法
- 统计新固件试启动次数
- 检测新固件是否完成确认
- 新固件失败时自动回滚旧固件

Bootloader 要尽量简单、稳定，原则上不要频繁升级。

## 5. OTA 状态定义

建议 OTA 状态区保存以下字段：

```c
typedef struct {
    uint32_t magic;
    uint32_t struct_version;

    uint8_t current_slot;       // 当前有效分区：A / B
    uint8_t target_slot;        // 待启动分区：A / B
    uint8_t previous_slot;      // 回滚分区：A / B
    uint8_t ota_state;

    uint32_t current_version;
    uint32_t target_version;

    uint32_t boot_try_count;
    uint32_t max_boot_try_count;

    uint32_t last_ota_result;
    uint32_t last_fail_reason;

    uint32_t firmware_size;
    uint8_t firmware_sha256[32];

    uint32_t crc32;
} ota_info_t;
```

OTA 状态建议如下：

| 状态 | 含义 |
|---|---|
| IDLE | 无升级任务，设备正常运行 |
| DOWNLOADING | 正在下载固件 |
| DOWNLOADED | 固件下载完成 |
| VERIFY_FAILED | 下载后的固件校验失败 |
| PENDING_VERIFY | 新固件待启动验证 |
| VALID | 当前固件已经确认有效 |
| ROLLBACK | 新固件失败，已回滚旧版本 |
| FAILED | 升级失败，但旧固件仍可运行 |

失败原因建议如下：

| 失败原因 | 含义 |
|---|---|
| NONE | 无失败 |
| LOW_BATTERY | 电量或供电不足 |
| DEVICE_BUSY | 设备正在工作 |
| HW_NOT_MATCH | 硬件版本不匹配 |
| DOWNLOAD_TIMEOUT | 下载超时 |
| DOWNLOAD_CRC_ERROR | 分包 CRC 校验失败 |
| FLASH_WRITE_ERROR | Flash 写入失败 |
| IMAGE_SIZE_ERROR | 固件大小不一致 |
| IMAGE_HASH_ERROR | SHA256 校验失败 |
| SIGNATURE_ERROR | 数字签名校验失败 |
| NEW_FW_BOOT_FAILED | 新固件启动失败 |
| NEW_FW_SELF_CHECK_FAILED | 新固件自检失败 |
| REPORT_TIMEOUT | 升级结果上报失败 |

## 6. LED 状态设计

建议 LED 指示尽量简单，不要设计得过于复杂，否则用户和售后都很难记。

| 设备状态 | LED 表现 |
|---|---|
| 正常运行 | 绿灯慢闪 |
| 网络异常 | 蓝灯慢闪 |
| 设备故障 | 红灯慢闪 |
| 准备升级 | 黄灯常亮 2 秒 |
| 下载固件中 | 黄灯慢闪 |
| 收到固件分包 | 黄灯短闪 |
| 校验固件中 | 黄灯快闪 |
| 准备重启 | 黄灯常亮 3 秒 |
| 新固件首次验证 | 蓝灯快闪 |
| 升级成功 | 绿灯快闪 3 次 |
| 升级失败 | 红灯快闪 5 次 |
| 自动回滚成功 | 红灯快闪 3 次，然后绿灯快闪 3 次 |

LED 属于 P1 能力，建议第一版做。  
如果硬件 LED 资源有限，至少保留 4 种状态：

| 最小状态 | LED 表现 |
|---|---|
| 正常运行 | 绿灯慢闪 |
| 升级中 | 黄灯慢闪 |
| 升级成功 | 绿灯快闪 3 次 |
| 升级失败或回滚 | 红灯快闪 5 次 |

## 7. 完整升级流程

当前 App 侧已按统一产品线 MQTT topic 实现 OTA：

```text
设备信息与 OTA 状态：SolarClean/devices/{sn}/status
App 下发 OTA 命令：SolarClean/devices/{sn}/control
```

下面示例主要说明 payload 语义；topic 以本节统一 topic 为准。

### 7.1 正常启动

设备上电后，Bootloader 读取 OTA 状态区。

如果当前状态为 `VALID` 或 `IDLE`：

```text
Bootloader 校验 current_slot 固件
校验通过后启动当前固件
校验失败则尝试启动另一个有效分区
```

App 固件启动后，上报：

```json
{
  "v": 1,
  "type": "deviceInfo",
  "ts": 8481,
  "hardware_version": "HW-A",
  "firmware_version": "1.0.3",
  "inner_version": 3,
  "slot": "A",
  "ota_status": "IDLE",
  "last_ota_result": "NONE",
  "last_fail_reason": "NONE",
  "network": "4g"
}
```

### 7.2 查询新版本

App 向服务器查询新版本。

请求示例：

```http
GET /api/data/appversion/getAppVersion?productId=3&type=2
```

服务器返回：

```json
{
  "id": 8,
  "version": "4",
  "productName": "光伏清洗",
  "techDesc": "光伏清洗bin",
  "innerVersion": "6",
  "publishDate": "2026-05-29 00:00:00",
  "path": "/download/SolarClean_APP_V0.0.1_0X08020000.bin",
  "type": 2
}
```

版本类型：

| type | 含义 | productId |
| --- | --- | --- |
| 1 | App 更新包 | 1=水枪控制，2=水桶控制 |
| 2 | 设备固件更新包 | 3=光伏清洗，4=消防吊桶 |

App 使用设备上报的内部版本 `inner_version` 与服务器 `innerVersion` 做本地比较。`firmware_version` 和服务器 `version` 主要用于用户展示。服务器 `path` 如果是相对路径，App 下发 `START_OTA` 前补全为完整下载地址。

App 显示：

```text
当前版本：1.0.3
最新版本：1.0.4
当前内部版本：3
最新内部版本：6
更新内容：修复电机保护逻辑
是否升级？
```

### 7.3 用户确认升级

用户在 App 点击升级后，App 下发 OTA 指令给设备。

指令示例：

```json
{
  "command": "START_OTA",
  "target_version": "4",
  "target_inner_version": 6,
  "file_size": 245760,
  "sha256": "xxxxxxxx",
  "download_url": "https://api.tjinnovations.cloud/download/SolarClean_APP_V0.0.1_0X08020000.bin"
}
```

服务器返回 `fileSize` 和 `sha256` 后，App 必须下发 MQTT 字段 `file_size` 和 `sha256`。单片机必须用这两个字段做固件大小和 Hash 校验。

设备收到指令后先检查升级条件：

- 电量是否足够
- 供电是否稳定
- 电机是否停止
- 水泵是否停止
- 清洗任务是否结束
- 设备是否处于待机状态
- 硬件版本是否匹配
- Flash 空间是否足够
- 当前是否已经在升级中

如果条件不满足，设备拒绝升级并上报原因。

示例：

```json
{
  "ota_status": "FAILED",
  "reason": "DEVICE_BUSY",
  "message": "设备正在执行清洗任务，暂不允许升级"
}
```

### 7.4 下载固件

设备进入 `DOWNLOADING` 状态。

下载规则：

- 下载到备用分区
- 每包下载后做 CRC 校验
- 每包写入 Flash 后确认写入成功
- 支持断点续传
- 支持超时重试
- 建议最多重试 3 次
- 下载过程中旧固件仍然在运行

进度上报：

```json
{
  "ota_status": "DOWNLOADING",
  "progress": 42,
  "downloaded": 103219,
  "total": 245760
}
```

如果下载失败：

```text
停止 OTA
保留当前运行固件
标记 ota_state = FAILED
记录 last_fail_reason
向 App / 服务器上报失败
```

下载失败不会导致设备变砖，因为当前旧固件还没有被替换。

### 7.5 固件校验

下载完成后进入校验阶段。

必须校验：

- 固件大小
- SHA256
- 数字签名
- 固件头
- 硬件版本
- 目标版本
- 分区地址范围

校验通过后，写入 OTA 状态区：

```text
previous_slot = A
target_slot = B
ota_state = PENDING_VERIFY
boot_try_count = 0
max_boot_try_count = 3
target_version = 1.0.4
```

然后设备上报：

```json
{
  "ota_status": "READY_TO_REBOOT",
  "target_version": "1.0.4"
}
```

如果校验失败：

```text
标记备用分区无效
保持旧固件继续运行
ota_state = VERIFY_FAILED
记录失败原因
上报失败
```

### 7.6 重启并试启动新固件

设备校验通过后主动重启。

Bootloader 启动后读取 OTA 状态区：

```text
发现 ota_state = PENDING_VERIFY
发现 target_slot = B
校验 B 区固件
boot_try_count += 1
尝试启动 B 区
```

如果 B 区固件校验失败：

```text
不启动 B 区
切回 previous_slot
ota_state = ROLLBACK
last_fail_reason = IMAGE_HASH_ERROR 或 SIGNATURE_ERROR
```

### 7.7 新固件首次启动验证

新固件第一次启动后，不允许立刻确认升级成功。

必须先进入试运行阶段，执行自检：

- 主循环是否正常运行
- 看门狗是否正常喂狗
- 参数区是否能读取
- 关键任务是否创建成功
- 电机驱动初始化是否正常
- 水泵控制初始化是否正常
- 传感器初始化是否正常
- 通信模块是否正常
- 能否连接服务器
- 设备型号和硬件版本是否匹配
- 旧配置是否兼容新固件

新固件上报：

```json
{
  "ota_status": "VERIFYING",
  "version": "1.0.4",
  "slot": "B"
}
```

建议验证时间：

```text
最短 30 秒
推荐 60 到 120 秒
```

### 7.8 确认升级成功

如果新固件自检通过，并且运行稳定，则新固件写入确认标志：

```text
current_slot = B
previous_slot = A
ota_state = VALID
boot_try_count = 0
last_ota_result = SUCCESS
last_fail_reason = NONE
```

然后上报服务器：

```json
{
  "ota_status": "SUCCESS",
  "firmware_version": "1.0.4",
  "slot": "B"
}
```

App 显示：

```text
升级成功
当前版本：1.0.4
```

### 7.9 新固件启动失败

如果新固件有问题，可能出现以下情况：

- 启动直接死机
- 看门狗反复复位
- 进不了主循环
- 通信模块起不来
- 参数区读取失败
- 电机或传感器初始化失败
- 自检失败
- 一直没有写入升级成功确认标志

Bootloader 在下次启动时检查：

```text
ota_state 仍然是 PENDING_VERIFY
boot_try_count 已超过 max_boot_try_count
```

则执行自动回滚：

```text
target_slot 标记无效
current_slot = previous_slot
ota_state = ROLLBACK
last_ota_result = FAILED
last_fail_reason = NEW_FW_BOOT_FAILED
启动旧固件
```

旧固件启动后上报：

```json
{
  "ota_status": "ROLLBACK",
  "current_version": "1.0.3",
  "failed_version": "1.0.4",
  "reason": "NEW_FW_BOOT_FAILED"
}
```

App 显示：

```text
升级失败，设备已自动恢复到旧版本
当前版本：1.0.3
失败版本：1.0.4
原因：新固件启动失败
```

## 8. 失败场景处理

| 场景 | 处理方式 | 是否影响旧固件 |
|---|---|---|
| 电量不足 | 拒绝升级并上报 LOW_BATTERY | 不影响 |
| 设备正在工作 | 拒绝升级并上报 DEVICE_BUSY | 不影响 |
| 硬件版本不匹配 | 拒绝升级并上报 HW_NOT_MATCH | 不影响 |
| 下载超时 | 重试，超过次数后失败 | 不影响 |
| 分包 CRC 错误 | 重新下载该包 | 不影响 |
| Flash 写入失败 | 停止升级并上报 FLASH_WRITE_ERROR | 不影响 |
| 固件大小错误 | 标记校验失败 | 不影响 |
| SHA256 错误 | 标记校验失败 | 不影响 |
| 签名错误 | 标记校验失败 | 不影响 |
| 新固件启动死机 | Bootloader 自动回滚 | 不影响 |
| 新固件自检失败 | 主动标记失败并重启回滚 | 不影响 |
| 成功但上报失败 | 本地记录结果，下次联网补报 | 不影响 |

## 9. App 显示建议

App 端建议显示以下状态：

```text
发现新版本
准备升级
正在下载 0% - 100%
正在校验
设备即将重启
正在验证新固件
升级成功
升级失败
已自动回滚到旧版本
```

失败时必须显示可理解的原因：

```text
电量不足，请充电后再升级
设备正在工作，请停止清洗任务后再升级
网络异常，固件下载失败
固件校验失败，已取消升级
新固件启动失败，设备已恢复旧版本
```

## 10. 服务器发布和回滚策略

由于本产品不是高频更新产品，服务器策略应以“可控发布”为主，不以“自动推送”为主。

第一版建议发布方式：

```text
1. 新固件先上传到服务器，但默认不公开
2. 先绑定内部测试设备可见
3. 内部设备升级验证通过后，开放给指定客户或指定设备
4. App 手动检测到新版本
5. 用户或售后确认后升级
6. 服务器记录升级结果
```

服务器第一版至少需要记录：

- 升级设备数量
- 成功数量
- 失败数量
- 回滚数量
- 失败原因分布
- 当前稳定版本
- 被禁用的问题版本

如果某个版本失败率过高：

```text
立即停止推送该版本
把该版本标记为 blocked
必要时发布修复版本
```

对于低频升级产品，建议服务器只重点维护：

```text
当前稳定版本
上一个稳定版本
正在测试的新版本
已禁用的问题版本
```

不建议第一版就做很多历史版本并存、复杂批量任务和自动强制推送。

## 11. 安全要求

第一版保命必须做：

- 固件 SHA256 校验
- 固件头包含硬件版本
- 固件头包含版本号
- 固件头包含文件大小
- 固件头包含合法 magic 标识

公网商用建议尽早做：

- HTTPS 下载
- 固件数字签名校验
- 服务器端版本黑名单
- 设备端拒绝硬件版本不匹配的固件

后期增强：

- 防止低版本恶意覆盖高版本，可根据需要做防回滚策略
- 固件加密
- 固件密钥轮换机制

建议固件头包含：

```c
typedef struct {
    uint32_t magic;
    uint32_t header_version;
    uint32_t firmware_version;
    uint32_t hardware_version;
    uint32_t image_size;
    uint8_t sha256[32];
    uint8_t signature[64];
    uint32_t header_crc;
} firmware_header_t;
```

## 12. 分阶段落地计划

### 12.1 阶段一：保命版 OTA

阶段一目标：先保证升级失败不会变砖。

必须完成：

- Bootloader
- A/B 双固件分区
- OTA 状态区
- 固件下载到备用分区
- 固件大小校验
- 固件 Hash 校验
- PENDING_VERIFY 状态
- 新固件启动确认
- 看门狗复位
- 启动失败自动回滚
- 基础失败原因记录

这个阶段可以暂时不追求体验，但必须保证：

```text
旧固件不会被直接覆盖。
新固件没有确认成功之前，旧固件始终可以恢复。
```

### 12.2 阶段二：低频商用版 OTA

阶段二目标：满足低频商用升级。用户、现场人员和售后能看懂升级过程，研发能定位问题。

建议完成：

- App 显示升级进度
- App 显示失败原因
- LED 显示升级状态
- 服务器记录升级结果
- 设备升级前检查电量/供电
- 设备升级前检查是否正在工作
- 硬件版本匹配
- 下载失败重试
- Flash 写入失败检测
- App 手动检测更新
- 用户或售后手动确认升级
- 服务器支持版本启用/禁用

这个阶段完成后，可以进入小批量试点。

### 12.3 阶段三：小范围维护版 OTA

阶段三目标：支持指定客户、指定设备、小范围维护升级。

建议完成：

- 指定设备可见新版本
- 指定客户可见新版本
- 版本黑名单
- 升级成功率统计
- 失败原因统计
- 按设备型号/硬件版本推送
- 问题版本停止发布
- 简单升级任务记录
- 失败设备列表导出

这个阶段适合设备数量开始增加，或者设备已经交付给多个客户后做。

### 12.4 阶段四：安全增强版 OTA

阶段四目标：提升安全性和长期运维能力。

建议完成：

- 固件数字签名
- HTTPS 下载
- 防回滚策略
- 固件加密
- 断点续传
- 蓝牙或串口兜底升级工具
- 工厂或售后刷机工具

以下能力暂不优先：

- 差分升级
- 多区域 CDN
- 自动大规模推送
- 复杂强制升级策略

如果设备已经连接公网，数字签名和 HTTPS 应提前做，不建议拖到太后面。

## 13. 一句话结论

光伏清洗设备 OTA 是低频维护能力，不是高频运营能力。

第一版重点不是“升级体验多高级”，而是“偶尔升级一次，也必须稳，失败也能回来”。

只要 A/B 双分区、Bootloader 回滚、固件校验、启动确认这几件事做好，即使新固件是坏的，设备也不会直接变砖，而是会自动恢复到旧版本，并向 App / 服务器上报失败原因。
