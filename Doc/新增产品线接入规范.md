# 新增产品线接入规范

本文是以后新增产品线时必须先读的总规范。它不写某个产品的具体需求，而是规定所有产品都要和当前 TJI Platform 的架构、UI、交互、MQTT、悬浮窗、测试方式保持一致。

例如后续要新增「六段抛投」，流程应该是：

1. 先读本文，确定平台统一规则。
2. 再在产品自己的文档里写「六段抛投」需求、协议字段、控制逻辑。
3. 最后按本文的目录、命名、UI、测试清单落代码。

---

## 1. 总原则

一句话：

> 公共层只做平台能力，产品层只做自己的业务。

公共层可以知道：

- 有哪些产品类型。
- 产品显示名称、标签、首页入口。
- 账号绑定设备列表。
- MQTT 连接、订阅、事件分发。
- 产品运行时统一快照。
- 公共 UI 组件、主题、悬浮窗框架。

公共层不应该知道：

- 某个产品私有字段。
- 某个产品控制命令的业务含义。
- 某个产品状态 payload 的详细结构。
- 某个产品页面里的具体控制组件。

产品层必须自己维护：

- `model`
- `mqtt`
- `repository`
- `runtime`
- `ui/control`
- `ui/floating`
- `ui/icon`
- `viewmodel`

---

## 2. 新产品目录规范

新增产品必须放在：

```text
app/src/main/java/com/tji/device/product/{productCode}/
```

推荐目录：

```text
product/{productCode}/
  model/          # 产品私有状态、控制参数、运行时模型
  mqtt/           # MQTT topic、入站解析、协议常量
  repository/     # 产品状态仓库、控制发布仓库、OTA 仓库（如需要）
  runtime/        # ProductRuntimeController 实现
  ui/
    control/      # 产品详情控制页
    floating/     # 产品悬浮窗面板
    icon/         # 产品图标、控制图标
  viewmodel/      # 产品控制 ViewModel
```

不要把产品私有模型放到：

```text
data/model/
ui/main/
ui/components/
```

除非它确实是跨产品公共能力。

---

## 3. ProductType 和 ProductDefinition

新增产品第一步：

文件：

```text
app/src/main/java/com/tji/device/data/model/Product.kt
```

需要新增：

- `ProductType`
- `ProductDefinition`
- `ProductCatalog.fromBackendFields(...)` 识别规则

示例：

```kotlin
enum class ProductType {
    FireBucket,
    SolarClean,
    FC100_FireDrop
}
```

`ProductDefinition` 必须包含：

- `displayName`：用户看到的中文名，如「六段抛投」。
- `shortLabel`：英文/短标签，如 `Dropper`。
- `description`：产品线说明。
- `platformSubtitle`：首页卡片副标题。
- `platformValueLine`：首页价值短句。

注意：

- 首页、悬浮窗、设置页不能到处硬编码产品名字。
- 产品名统一从 `ProductCatalog.definitionOf(productType)` 取。
- 设备名优先使用后台绑定设备名或 MQTT 运行时名称；不要用产品名冒充设备名。

## 4. UI 风格规范

当前 App 风格是现代 SaaS 企业移动端风格。

统一视觉方向：

- 背景：浅灰 `#F5F7FA`。
- 卡片：白色到 `#FAFBFF` 微渐变。
- 主色：`#1677FF`。
- 圆角：主要卡片 12dp 起，首页大卡片可 22-28dp。
- 阴影：轻量、干净，不做重阴影。
- 状态色必须清晰，不要柔化到看不清。

状态色：

```text
在线 / 成功：#52C41A
警告：#FA8C16
离线 / 错误：#FF4D4F
执行中 / 主操作：#1677FF
```

优先使用现有 token：

```text
ui/theme/Color.kt
ui/theme/Type.kt
ui/theme/Theme.kt
```

不要在新页面大量写：

```kotlin
Color(0xFF...)
fontSize = 17.sp
```

如果确实需要新增颜色或字号，先考虑是否应该变成全局 token。

## 5. 公共 UI 组件复用

新增产品页面必须优先复用：

```text
ui/components/TjiCards.kt
ui/components/TjiButtons.kt
ui/components/TjiControlSlider.kt
ui/components/TjiFeedbackBadge.kt
ui/components/TjiMiniSwitch.kt
```

常用组件：

- `TjiCardShell`：普通卡片容器。
- `TjiSectionCard`：带标题的控制/数据卡片。
- `TjiActionButton`：控制按钮。
- `TjiControlSlider`：控制滑动条。
- `TjiMiniSwitch`：悬浮窗小开关。
- `TjiFeedbackBadge`：ACK 成功/失败提示。
- `TjiOnlineStatus`：在线/离线状态。

不要每个产品自己复制一份：

- 卡片阴影。
- 在线离线圆点。
- ACK 胶囊。
- 控制按钮。
- 滑动条。
- 小 Switch。

如果现有公共组件不够用，优先扩展公共组件，而不是产品里再造一套。

## 6. 首页和产品入口

首页只展示产品入口，不展示某个产品的复杂控制。

新增产品需要接入：

- 产品卡片。
- 产品图标。
- 产品背景图。
- 产品设备列表。

相关位置：

```text
ui/main/MainScreen.kt
ui/components/AppVisualAssets.kt
ui/icon/product/ProductIconRouter.kt
```

首页产品卡片规范：

- 左侧文字、图标。
- 右侧产品场景图。
- 图片必须有横向渐变遮罩，不能影响文字可读性。
- 横屏可以两列，移动端竖屏单列。

设备列表规范：

- 进入产品后再展示该产品下的设备。
- 一个设备卡片一行主要信息：图标、设备名、在线状态、箭头。
- 设备号作为第二行补充；如果设备名和设备号一样，可以只显示一个。
- 不要在用户界面展示开发说明、控制线、协议提示等内部文字。

---

## 7. 控制详情页规范

新增产品必须有自己的控制页：

```text
product/{productCode}/ui/control/{Product}ControlScreen.kt
```

控制页推荐结构：

```text
顶部栏：返回 + 设备名 + 在线状态 + 设置入口（如需要）
设备控制：主要操作放最上面
设备状态：数据展示放下面
设备设置：SN、版本、OTA 等放设置页
```

控制页不要展示：

- 协议说明。
- 下一阶段开发提示。
- MQTT topic。
- payload 字段。
- “暂无接入协议”等开发文案。

这些内容应该写进 `Doc`，不能给客户看到。

---

## 8. 悬浮窗规范

新增产品如果需要悬浮窗，必须实现自己的面板：

```text
product/{productCode}/ui/floating/{Product}FloatingPanel.kt
```

悬浮窗框架仍然走公共层：

```text
ui/floating/
```

产品只负责面板内容，不要改悬浮窗 Service 的基础逻辑，除非是全产品通用需求。

悬浮窗必须遵守：

- 进入哪个设备，悬浮窗显示哪个设备。
- 标题显示设备名或 SN，不能显示产品名冒充设备名。
- 离线时控制不可点击。
- ACK 反馈只显示客户能懂的文案，如「水泵设置成功」「抛投失败」。
- 不显示原始 ack、cmd、payload。
- 不重复订阅 MQTT；悬浮窗只读取已有运行时状态并发布控制。

多子设备产品参考消防吊桶：

- 一个账号可以有多个 Link。
- 一个 Link 下可以有多个子设备。
- 悬浮窗进入某个 Link 后，只切换该 Link 下的子设备。
- 控制命令发给当前 Link，不直接发给子设备，除非该产品协议明确要求。

## 9. MQTT 协议规范

新产品默认使用统一 topic 结构：

```text
{ProductCode}/devices/{deviceSn}/lifecycle
{ProductCode}/devices/{deviceSn}/status
{ProductCode}/devices/{deviceSn}/control
```

必须在产品包内定义：

```text
product/{productCode}/mqtt/{Product}MqttTopics.kt
product/{productCode}/mqtt/{Product}MqttInbound.kt
```

并接入公共路由：

```text
service/mqtt/MqttTopicLayout.kt
service/MqttEventHandler.kt
di/AppContainer.kt
```

控制命令规范：

- App 下发 `cmd` 数字命令码。
- `cmdName` 只用于日志和 MQTTX 调试。
- MCU / 设备端必须按 `cmd` 分发。
- 每条控制命令建议有 `msgId`。
- 设备收到控制后应返回 `type=ack`。

ACK 展示规范：

- App 不展示原始 ACK。
- App 根据 `ok=true/false` 显示「具体操作 + 设置成功/失败」。
- 超时也要有用户可理解提示。

在线离线规范：

- `lifecycle` 负责 online/offline。
- `status state` 负责周期状态。
- 如果使用 retained state，App 必须判断 `ts` 新鲜度，不能只因为收到 retained 就认为在线。

## 10. Runtime 接入规范

每个产品必须实现：

```text
product/{productCode}/runtime/{Product}RuntimeController.kt
```

输出统一快照：

```kotlin
ProductDeviceRuntimeSnapshot(
    serialNumber = ...,
    name = ...,
    productType = ...,
    isOnline = ...,
    childCount = ...,
    payload = ...
)
```

注意：

- `payload` 可以放产品私有运行时对象。
- 公共首页只能读 `serialNumber/name/productType/isOnline/childCount`。
- 产品详情页可以把 `payload` 转回自己的模型。

然后注册到：

```text
di/AppContainer.kt
ProductRuntimeRegistry(...)
```

## 11. 依赖注入规范

当前项目使用手写 DI：

```text
di/AppContainer.kt
```

新增产品通常需要注册：

- Repository。
- MQTT inbound。
- RuntimeController。
- ViewModelFactory。
- FloatingQuickControl 或 FloatingPanel 相关依赖。

不要在 Composable 或 ViewModel 里直接 `new` Repository / MQTT 管理器。

推荐：

```kotlin
class XxxViewModel(
    private val repository: XxxRepository
) : ViewModel()
```

然后通过 Factory 和 `AppContainer` 注入。

## 12. 图标和素材规范

产品图标放在：

```text
product/{productCode}/ui/icon/
```

公共图标放在：

```text
ui/icon/common/
```

产品入口图标接入：

```text
ui/icon/product/ProductIconRouter.kt
```

要求：

- 尽量用 Compose `ImageVector` 代码图标。
- 图标需要有 Preview。
- 同一产品图标风格要统一。
- 不要用 emoji。

产品背景图接入：

```text
ui/components/AppVisualAssets.kt
```

图片建议：

- 首页产品卡片图：横向构图，主体靠右。
- 建议宽度 1200-1600px。
- 高度 600-900px。
- 保证右侧主体完整，左侧可被渐变遮罩覆盖。

## 13. 产品文档规范

每个具体产品必须有自己的需求文档。

推荐位置：

```text
Doc/products/{productCode}/README.md
Doc/products/{productCode}/protocol.md
Doc/products/{productCode}/mcu.md
Doc/products/{productCode}/server.md
Doc/products/{productCode}/app.md
```

如果暂时不想建多文件，至少建：

```text
Doc/products/{productCode}/README.md
```

产品文档需要写清：

- 产品是什么。
- 设备 SN 从哪里来。
- MQTT topic。
- lifecycle/status/control payload。
- cmd 命令码表。
- ACK 规则。
- 在线离线规则。
- App 控制项。
- 悬浮窗控制项。
- OTA 是否需要。
- MCU 要做什么。
- 服务器要做什么。
- App 要做什么。
- 当前不做什么。

## 14. 测试规范

新增产品至少要补这些测试：

### 单元测试

路径：

```text
app/src/test/java/
```

建议覆盖：

- 产品类型识别。
- MQTT payload 解析。
- ACK 文案映射。
- 在线离线判断。
- 悬浮窗选中设备逻辑。
- 多子设备切换逻辑。

### 手动回归

必须验证：

- 登录后能看到产品入口。
- 进入产品页后只显示该产品设备。
- 点击设备后进入正确控制页。
- 悬浮窗显示当前设备，不串到上一个产品。
- 离线设备不可控制。
- 控制命令发出后能看到 ACK 结果。
- 横屏和小屏可用。

### MQTT 联调

必须验证：

- App 订阅正确 topic。
- 设备发布 lifecycle/status 后 App 状态正确。
- App 发布 control 后设备能收到。
- 设备返回 ack 后 App 显示正确提示。
- 断电/断网后 App 能变离线。

## 15. 新产品接入清单

新增产品时按这个顺序做：

1. 在 `ProductType` 增加产品类型。
2. 在 `ProductCatalog` 增加 `ProductDefinition`。
3. 增加产品图标和场景图映射。
4. 建产品目录 `product/{productCode}`。
5. 建产品私有 `model`。
6. 建 MQTT topic 和 inbound。
7. 建 repository。
8. 建 runtime controller，并注册到 `ProductRuntimeRegistry`。
9. 建控制页，并接入 `ProductControlRoute`。
10. 建悬浮窗面板，并接入公共悬浮窗分流。
11. 在 `AppContainer` 注册依赖。
12. 写产品文档。
13. 补单元测试。
14. 做真实 MQTT 联调。
15. 做横屏、小屏、离线、弱网回归。

---

## 16. 不要做的事

不要：

- 把新产品状态塞进消防吊桶模型。
- 把某个产品 ViewModel 做成公共 ViewModel。
- 在首页直接解析产品私有 payload。
- 在悬浮窗里重复订阅 MQTT。
- 把测试设备硬编码进 release。
- 在客户 UI 上展示协议说明、topic、payload、下一阶段开发提示。
- 为了临时能跑，把产品名当设备名显示。
- 不写 ACK，只让用户猜控制有没有成功。

---

## 17. 六段抛投接入示例方向

如果后续新增「六段抛投」，建议产品代码名使用：

```text
FC100_FireDrop
product/firedrop
```

实际命名需要和后台、MCU、产品文档统一后再定。

它应该单独写：

```text
Doc/products/dropper/README.md
```

里面再定义：

- 六段分别是什么。
- 是否一个设备有 6 个通道。
- 控制命令是按通道编号，还是按阶段编号。
- 是否需要安全确认。
- 是否需要一次性全部复位。
- 是否需要悬浮窗快捷控制。
- 是否需要 OTA。

这些是产品自己的需求，不应该写进本文。
