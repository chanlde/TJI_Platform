# ADR-001: 拆分 Doc 根目录并采用领域化文档结构

## Status

Accepted

## Context

`Doc/` 根目录原来同时放置平台架构、类职责、OTA、设备绑定、UI/UX、服务器接口、产品线说明和人工 review 清单。随着消防吊桶、光伏清洗、六段抛投、喊话器、无线电侦测等产品线增多，单层目录会导致三类问题：

- 新成员不知道先读哪一份文档。
- 新产品线文档容易继续堆在根目录。
- 文件名中英混用，链接和检索成本偏高。

## Decision

将 `Doc/` 拆分为以下领域目录：

- `architecture/`：平台架构、类职责、产品接入规范和 ADR。
- `features/`：具体功能方案、UI/UX 优化方案。
- `features/ota/`：OTA 专项方案。
- `protocols/`：平台级或跨产品协议说明。
- `product-lines/`：单产品线自己的需求、协议、MCU、服务端和 App 文档。
- `operations/`：自动化测试、服务端待办、人工 review 清单。

文件名采用英文 kebab-case，文档标题继续使用中文。

## Alternatives Considered

- **继续平铺在 `Doc/` 根目录**：改动最小，但会继续扩大查找成本。
- **按产品线全部拆分**：产品文档更集中，但平台级架构、UI/UX、OTA 和服务端待办不自然。
- **直接改为 `docs/` 并引入文档站**：更标准，但当前需求是轻量整理，文档站会增加维护成本。

## Consequences

- Positive: 文档入口更清楚，新产品线有固定位置，路径更适合长期链接。
- Positive: 本轮新增架构文档和迁移表有独立目录，不污染平台总架构。
- Negative: 历史链接需要一次性更新。
- Negative: 旧文档标题仍然保留历史上下文，后续还需要逐步补状态和归档策略。

## Trade-offs

本轮优先解决“找得到、放得对、链接稳定”，暂不做大规模内容重写。过期文档删除需要业务确认，因此只删除明确无用的系统文件。
